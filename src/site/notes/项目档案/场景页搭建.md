# 场景页搭建

> 用 Claude Code 做了静态参考页（HTML 原型），配合技术（@鹏飞）和设计（@江江）团队，最终把「按场景选灯」的专题页推上线。

---

## 这是什么

在 Sealight 独立站上搭建了一个「按场景选灯」的专题页：[https://sealight-led.com/collection/by-scenario](https://sealight-led.com/collection/by-scenario)

用户不用懂灯泡型号，只需要告诉网站「我经常开夜路」「我住的地方雨雾多」「我想翻新老车外观」，系统就推荐对应的灯。

---

## 为什么这么做

原来 Sealight 网站是按产品系列组织的（S1、S2S、S7S…），对刚接触车灯的用户来说，一看就懵：这些型号有什么区别？我该买哪个？

我们之前做的用户调研（835 份问卷 + 四大分众）已经明确了四类典型用户：

| 人群 | 占比 | 核心需求 |
|------|------|---------|
| 高风险升级派 | 40.5% | 安全，恶劣天气穿透力 |
| 中低风险颜值党 | 25% | 外观翻新，色温好看 |
| 刚需小白 | 19.9% | 告诉我买哪个就行 |
| 技术精明党 | 14.6% | 参数对比，技术细节 |

场景页就是把这四类人的需求翻译成三个场景入口：夜路 / 恶劣天气 / 翻新外观。

---

## 怎么做的

我的角色是用 Claude Code 做**静态参考页**（HTML + CSS 原型），给技术和设计团队当「活的需求文档」。最终上线由技术团队在 Shopify 上实现。

整个过程分四步：

### 第一步：竞品调研

翻了 AUXBEAM、Lasfit、Morimoto 等品牌的场景页做法，看了他们的：
- 怎么分场景（按天气 / 按路况 / 按车型）
- 怎么呈现产品（卡片式 / 对比表 / 场景实拍）
- 怎么讲故事（用户痛点 → 解决方案 → 产品推荐）

输出了一份竞品场景营销研究报告。

### 第二步：场景划分 + 原型

确定了三个场景：
1. **Night Driving**（夜路驾驶）— 乡村道路、无路灯高速、长途通勤
2. **Storm Weather**（恶劣天气）— 大雨、浓雾、雪天
3. **Refresh Your Ride**（翻新爱车）— 老车卤素灯升级

用 HTML + CSS 做了可交互原型，包含：
- Hero 区（一句话主张 + 场景选择器）
- 每个场景的产品推荐卡片
- 适配查询（Fitment Finder）
- FAQ 折叠区

### 第三步：文案撰写

和 Claude 反复打磨了每个场景的文案，核心原则：
- 不讲 raw lumens（原始流明），讲 usable light（实际可用光）
- 不讲技术参数，讲驾驶体验
- 每个场景有独立的视觉语言和产品推荐

详见 [[项目档案/sealight-场景页文案\|场景页文案]]。

### 第四步：技术对接 + 上线

我把静态原型 + 文案 + 需求文档交给技术团队（@鹏飞），他们在 Shopify 上实现了真正的场景页。设计（@江江）在原型基础上做了视觉优化。最终上线到 `/collection/by-scenario`。

> 我做的是「参考页」——让技术和设计能看到效果、理解意图。真正把页面做上线、做到能跑、能卖货的，是技术团队。

---

## 踩过的坑

- **第一次原型太复杂了**：做了四个独立页面（Night / Storm / Offroad / Refresh），结果发现维护成本太高，推倒重来做成三合一单页
- **文案改了无数版**：英文文案的「接地气」和「专业感」之间平衡很难，改到第 4 版才定稿
- **适配查询是硬需求**：从问卷数据看，刚需小白最在意的就是「告诉我买哪个」，Fitment Finder 的位置和交互打磨了很久

---

## 学到了什么

- **用户分层是场景化的基础** — 没有前面的 835 份问卷和聚类分析，场景划分就是拍脑袋
- **原型比文档有用** — 给开发看 HTML 原型比写 10 页需求文档都清楚（但原型 ≠ 上线，真正的工程实现靠技术团队）
- **英文文案要多看竞品** — 竞品的用词、句式、情绪调性是最好老师
- **AI 做执行，人做判断** — Claude 可以帮我改文案、写 CSS、做竞品分析，但「这个场景到底该不该有」「这句话用户看了会不会下单」只有人能判断
